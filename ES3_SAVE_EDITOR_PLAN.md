# 在线 ES3 游戏存档修改器开发方案

版本：v0.1.0  
日期：2026-06-06  
项目代号：Game Assistance

## 1. 项目目标

开发一个浏览器端运行的在线游戏存档修改器，允许用户选择本地 `.es3` 存档文件，解析为可编辑结构化数据，用户修改后重新生成并下载新存档。

核心定位不是“某一个游戏的修改器”，而是一个面向 Unity Easy Save 3 存档的通用解析与编辑框架，并允许后续通过游戏配置模板增强特定游戏体验。

## 2. 可行性结论

可以实现，但必须接受以下边界：

1. `.es3` 常见于 Unity Easy Save 3，但它不是所有游戏都遵守的统一公开存档标准。
2. Easy Save 3 可以使用 JSON 等格式保存数据，也可以启用 AES 加密、Gzip 压缩或自定义类型序列化。
3. 未加密、未压缩或可自动识别压缩的 `.es3`，可以在浏览器端解析、编辑、重新导出。
4. 加密 `.es3` 需要用户提供密码或游戏模板内置解密参数，否则只能提示无法解析。
5. 自定义复杂类型可以尽量以通用 JSON 树编辑器呈现，但无法保证自动理解所有字段含义。

因此本项目的正确目标是：

- 通用解析器：尽可能打开更多 ES3 存档。
- 通用编辑器：先让用户能安全修改结构化字段。
- 游戏模板系统：让热门游戏拥有更友好的表单、字段中文名、范围校验和一键修改能力。

## 3. 参考信息

- savefile.space 当前是按游戏提供存档编辑入口，并非完全通用编辑器。
- Easy Save 3 官方文档说明它支持 AES 加密，默认不启用加密。
- Easy Save 3 官方文档说明压缩默认不启用，但可以通过设置启用。
- ES3 文件扩展名通常关联 Unity Easy Save 3，并可能以 JSON 形式保存游戏状态。

## 4. 产品范围

### v0.1 最小可用版本

目标：做出一个可以实际处理简单 `.es3` 文件的本地网页工具。

功能：

- 上传 `.es3` 文件。
- 自动识别文本 JSON。
- 自动识别 UTF-8 / UTF-8 BOM。
- 解析成功后展示字段树。
- 支持编辑 string、number、boolean、null。
- 支持对象和数组展开折叠。
- 保存时生成新的 `.es3` 文件下载。
- 不上传服务器，所有处理在浏览器本地完成。
- 修改前提示用户备份原存档。

不做：

- 不承诺解析加密存档。
- 不承诺理解字段含义。
- 不做云端保存。
- 不做账号系统。

### v0.2 解析增强版本

功能：

- 自动探测 Gzip / Deflate 压缩。
- 增加十六进制预览。
- 增加原始文本编辑模式。
- 增加 JSON 格式化、压缩输出。
- 增加解析错误定位。
- 增加修改 diff 预览。

### v0.3 游戏模板版本

功能：

- 新增游戏模板系统。
- 模板定义字段路径、显示名称、类型、最小值、最大值、枚举选项、说明。
- 对匹配模板的存档展示专用表单。
- 未匹配字段仍可在通用树编辑器中编辑。
- 支持模板导入导出。

### v0.4 加密与高级兼容版本

功能：

- 支持用户输入 ES3 AES 密码尝试解密。
- 支持模板携带非敏感解析策略。
- 支持保留原始文件结构元数据。
- 支持更严格的写回一致性检查。

注意：不内置商业游戏的破解密钥，不绕过网络游戏反作弊，不提供破坏性用途指导。

## 5. 技术路线

### 前端

首版使用单 HTML 文件实现：

- HTML
- CSS
- JavaScript
- TailwindCSS CDN

后续如功能复杂，再迁移到 Vite + TypeScript + React 或 Vue。

首版选择单文件的理由：

- 方便快速验证解析逻辑。
- 不需要构建工具。
- 用户可以直接打开使用。
- 适合作为原型和演示版本。

### 文件处理

浏览器 API：

- `FileReader` 或 `Blob.arrayBuffer()`
- `TextDecoder`
- `Blob`
- `URL.createObjectURL`

解析流程：

1. 用户选择文件。
2. 读取 ArrayBuffer。
3. 检测文件头。
4. 尝试按文本解码。
5. 尝试 JSON.parse。
6. 成功后生成可编辑数据模型。
7. 用户修改。
8. JSON.stringify 写回。
9. 生成下载文件。

### 数据模型

内部维护：

- `originalFileName`
- `originalBytes`
- `decodedText`
- `parsedData`
- `editedData`
- `parseMode`
- `warnings`
- `errors`

### 编辑器设计

通用树编辑器节点类型：

- object
- array
- string
- number
- boolean
- null

编辑原则：

- object / array 先只允许展开查看，不在 v0.1 里做任意新增删除。
- string / number / boolean 允许直接编辑。
- number 编辑必须校验是否为有效数字。
- 保存前重新校验整棵树。

## 6. UI 约束

首页就是工具本体，不做营销落地页。

布局：

- 顶部：产品名、状态、操作按钮。
- 左侧：文件信息、解析状态、警告、模板区域。
- 右侧：字段树编辑器。
- 底部或右上：下载修改后存档。

视觉风格：

- 工具型界面，密度适中。
- 不使用夸张 hero。
- 不使用大面积单色紫蓝渐变。
- 重要风险提示清楚但不打断流程。
- 移动端可用，但首要保证桌面端编辑效率。

## 7. 安全与合规边界

1. 默认本地处理，不上传用户存档。
2. 明确提示用户备份原文件。
3. 不提供联网游戏作弊、绕过反作弊、破解加密密钥等能力。
4. 对加密文件只支持用户自行提供密码。
5. 模板系统不内置敏感密钥。
6. 下载文件名默认添加 `_modified`，避免覆盖原文件。

## 10. 主要风险

### 风险 1：ES3 格式不统一

应对：把首版定义为 JSON 风格 ES3 编辑器，逐步扩展压缩、加密和模板能力。

### 风险 2：写回后游戏不认

应对：

- 尽量保留原始数据类型。
- 不自动重排非必要字段。
- 首版提示用户备份。
- 后续增加写回前校验和 diff。

### 风险 3：字段含义不可读

应对：通用编辑器负责可编辑，模板系统负责好用。

### 风险 4：大文件性能

应对：

- v0.1 先设定建议文件大小上限。
- 后续使用虚拟滚动和懒渲染树节点。

## 12. 《吾今有世家》模板记录

资料来源：

- `GameData.es3`
- `吾今有世家全解.xlsx`
- `存档信息.jpg`
- `游戏物品道具大百科.html`
- `游戏角色属性一览表.html`
- `游戏书籍属性一览表.html`
- 用户实测确认

已确认字段：

- `CGNum.value[0]`：铜钱数量。与快捷面板中的家族基础铜钱数一致。
- `CGNum.value[1]`：元宝数量。与快捷面板中的家族基础元宝数一致。
- `FamilyData.value[1]`：家族姓氏。
- `FamilyData.value[2]`：家族等级。
- `FamilyData.value[3]`：经验值。
- `FamilyData.value[4]`：家族对朝廷的影响。
- `FamilyData.value[5]`：仓库剩余容量。
- `FamilyData.value[8]`：是否皇室出家，`0` 为否，`1` 为是。
- `Time_now.value`：日期数组，结构为 `[年, 月, 日]`。
- `Prop_have.value`：物品数组，每项为 `[物品ID, 数量]`。
- `Horse_Have.value`：马匹数组，字段为 `[花色, 年龄, 力量, 速度, 智商, 主人/人物编号, 马匹信息]`。

模板化要求：

- 已确认字段在 UI 中使用明确中文名。
- 未确认字段保留 `槽位 n`，不强行解释。
- `吾今有世家全解.xlsx` 的 `物品代码` sheet 可作为物品 ID 字典来源。
- `游戏物品道具大百科.html` 已提供完整 `0-284` 物品字典，当前模板优先使用该完整数据。
- `游戏书籍属性一览表.html` 已提供 111 条书籍属性数据，可用于后续物品筛选和角色培养辅助。
- `游戏角色属性一览表.html` 已提供 17 条角色属性解释，可用于后续成员字段拆解。
- 该模板只作为《吾今有世家》的补充说明，不进入通用 ES3 解析核心。
